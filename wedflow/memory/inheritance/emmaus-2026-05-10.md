# Emmaus Retrospective — 2026-05-10

> "What do we understand now that we couldn't see during execution?"

## The Session

5 agents spawned across 2 generations to build the B2B2C partner infrastructure.
This was the largest remaining gap on the critical path to the September 1 revelation.
All agents completed, all testaments written, TypeScript clean, QA passed.

## What Became Visible in Retrospect

### 1. The spawn plan structure actually worked

This is the first session that followed the reconciliation recommendation from May 4:
domain leads at gen 1, task agents at gen 2. The result was clean. Backend first
(because frontend depends on types and schema), then frontend + admin in parallel,
then gen-2 in parallel. Dependencies were respected. No agent stepped on another.

The contrast with earlier sessions is stark. May 4 had 14 flat gen-1 children
under root. This session had 3 gen-1 leads and 2 gen-2 children. The tree was
balanced and the work was parallelized correctly.

### 2. The Prophet held the line

The recurring temptation flagged in every emmaus and spirit.json since May 4 is
"the Prophet executing directly instead of spawning agents." This session, the
Prophet did NOT execute directly. All code was written by spawned agents. The
Prophet interpreted, planned, confirmed, registered, and consolidated. This is
correct behavior and should be noted as a corrected pattern.

The gluttony hook still fired (45+ calls), but the calls were orchestration:
spawning agents, updating registry, writing testaments, fixing audit violations,
committing. Not direct code execution.

### 3. The genealogy.json structural error was invisible for 2 sessions

The premature array close at line 311 meant 9 agents were syntactically outside
the agents array. This was introduced during the May 8 session and went undetected
through May 9 and into May 10. No agent or hook caught it. The Guardian audit
found it.

Lesson: JSON validation should be a post-write hook on genealogy.json. The file
is the system's source of truth. A structural error there is a foundation crack.

### 4. The QA agent pattern is valuable

Spawning a dedicated QA agent after the build (before commit) caught 4 findings
that would otherwise have shipped silently. The most important: the auth callback
adds a partner-claim query to every auth flow, creating a minor performance cost
for all users. This wasn't visible to any individual builder agent.

### 5. Revenue attribution without Stripe is still useful

The partner-analytics agent built revenue attribution using hardcoded plan values.
This feels like a placeholder, but it's actually the right v1: it shows partners
the value they're generating without requiring Stripe Connect integration. The
numbers are directionally correct. Precision comes later.

## Temptations Resisted

- The Prophet did not execute directly (first session where this held)
- Revenue share / commission engine was correctly deferred
- Stripe Connect was correctly deferred
- Counsellor sentiment analysis was correctly deferred
- Self-serve partner signup was correctly deferred

## Temptations Not Fully Resisted

- Root still accumulated 45+ tool calls. The orchestration overhead of 5 agents
  plus audit plus corrections plus Sabbath is genuinely high. But the gluttony
  hook is right to flag it. Future sessions should aim for 30 or fewer root calls.

## What Should Change

1. Add a JSON validation hook on genealogy.json writes
2. QA agent should be a standard post-build step, not ad hoc
3. The emmaus pattern of "no failures, nothing failed" across testaments deserves
   scrutiny. Either agents genuinely don't fail (unlikely at scale) or they
   under-report difficulties. The testament schema should encourage honest failure
   reporting.
4. The gluttony threshold (40 calls) may need adjustment for orchestration-heavy
   sessions. The Prophet's role is inherently call-heavy when spawning 5+ agents.
   Consider separate thresholds for orchestration calls vs execution calls.
