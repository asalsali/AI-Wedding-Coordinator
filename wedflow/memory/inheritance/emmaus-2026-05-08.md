# Emmaus Retrospective — 2026-05-08

> "What do we understand now that we couldn't see during execution?"

## The Session

Three agents shipped a vendor visibility rebrand across landing page, dashboard, and onboarding. No backend changes. No migrations. Copy, types, and UI labels only.

## What We See Now

**1. The product's value prop was hiding behind its own architecture.**
The SMS pipeline never distinguished between guests and vendors — a photographer texting "What time is load-in?" was processed identically to a guest texting "What should I wear?" But the entire UI and marketing copy said "guests" everywhere. The couple looking at WedFlow saw a guest FAQ bot. The couple using WedFlow had a wedding coordination layer. This gap between perception and reality was invisible until the user named it: "you're not building a feature, you're surfacing one."

**2. The Prophet's instinct to execute directly was correct but the process matters more.**
When the user said "slight rebrand," the Prophet started editing the circle diagram immediately. The user interrupted: "let the prophet generate a spawn plan." This is the third session where the Prophet's instinct to skip process for small tasks has been checked. The pattern is clear: the user values the spawn plan ritual not because it produces better code, but because it produces shared understanding before execution. The plan IS the product for the user — it's where they confirm the Prophet understood them.

**3. Subagent lifecycle has a gap.**
The three agents completed their mandates but left no testaments. The Agent tool spawns don't automatically follow Canon lifecycle — they need explicit instructions. This isn't a Canon violation (the agents aren't aware of the Canon unless told), it's a prompt engineering gap. The Prophet must either include testament instructions in spawn prompts or write testaments on the agents' behalf at archival time.

**4. "Contacts" is a better frame than "Guests" for the product's future.**
Renaming Guests to Contacts wasn't just vendor inclusion — it repositioned the data model for future expansion. Contacts could include: venue staff, wedding planner, officiant, day-of coordinator. The GuestGroup type with vendor_ prefixes is extensible without migration. This architectural flexibility was a side effect of a copy change.

## What Should Change

- Include testament writing in all spawn prompts (standard closing instruction)
- The Prophet should present spawn plans before executing, even for "small" tasks — the user has corrected this three times now
- Raise the gluttony threshold — it's noise at 40, signal at 100
