# Emmaus Retrospective — 2026-05-04

> "What do we understand now that we couldn't see during execution?"

## What became clear only in retrospect

1. **The Prophet's temptation is structural, not incidental.** It happened 4+ times despite corrections. The temptation is strongest when fixes seem small or when prior agents hit permission walls. The framework needs a mechanical check, not just a disposition. The current Canon warns about it but has no enforcement mechanism. A pre-execution hook that checks "am I registered in genealogy?" before any Edit/Write would catch this.

2. **Shared number routing was always the right beta architecture.** The per-couple number model was premature optimization. 10-12 couples don't need 10-12 numbers. The pivot was easy because the pipeline was already well-separated (webhook resolves coupleId, everything downstream is coupleId-agnostic). Good architecture accidentally enabled a business model change.

3. **Mobile testing on the actual device is non-negotiable.** Every round of "it should work" was followed by "it doesn't work" when tested on the phone. The browse tool at 375x812 catches layout issues but misses interaction problems (composer clipping, touch target feel, scroll behavior).

4. **The user cares about the framework as much as the product.** Process compliance is not overhead to this user — it's a feature. The framework's governance value was validated by the user repeatedly insisting on proper Canon adherence even when it slowed things down.

5. **Synthesis produced a genuinely useful agent.** The inbox-composer (synthesis of interaction-fixer + mobile-architect) wired auto-draft pre-fill that neither parent had. The emergent skill was real, not theoretical. This validates the synthesis mechanism.

6. **Agent permission walls are the biggest Canon compliance threat.** When agents can't Write or run Bash, the work falls back to the Prophet, who then executes directly and bypasses the framework. Solving the permission model would solve most Canon violations.
